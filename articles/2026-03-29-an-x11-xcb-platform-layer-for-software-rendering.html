I. A Short History of Graphical Platforms

Long, long ago, in 1973 to be precise, was born what we now remember as the first computer to truly enable Graphical User Interfaces.

The Xerox Alto.

Named after the Xerox Palo Alto Research Center, or Xerox PARC for short, and building on concepts demonstrated by Doug Engelbart during the Mother of All Demos, it already features the core elements of what we take for granted in modern graphical systems: the 3-button mouse we know and love, the keyboard with its iconic QWERTY layout, and even windows within the SmallTalk runtime.

"What kind of windows?" you may ask; well, windows that overlap each other, one for each app, resizable, with menus and scrollbars. These ideas paved the way for the Desktop Metaphor, which still rules modern consumer OSes to this day.

The Xerox Alto in Action.

And with the big picture laid out, let's dive deeper into the inner workings of this legendary beast.

A. Inputs

The mouse and keyboard send hardware interrupts. These interrupts are translated into input events, such as a mouse motion or a key press, and the events are then stored into a queue. After computing which window owns which pixels - a step known as hit testing - events are dispatched to the appropriate window.

B. Outputs

Graphical applications write pixel values to a dedicated region of memory called the frame buffer. The video controller then scans that frame buffer, and translates the pixel values to instructions that control an electron gun. The gun shoots electrons at the Cathode Ray Tube (CRT) display, from left to right then going back left and one pixel down, in a zig-zag pattern called raster. When the electron beam hits the screen, a black pixel appears on the monochrome display. Apps that want to be free of visual artifacts must sync rendering with the electron beam's raster.

Note that the CPU writes to the frame buffer directly, like it does any other region of memory.

C. Windows

With multiple windows arises the issue of sharing a single screen, and a single frame buffer. At first, windows do not overlap, but instead tile the screen with the underlying frame buffer shared according to the tiles' boundaries.

The SmallTalk runtime brings support for overlapping windows, and it keeps track of which window owns which region of the screen. When a window is moved, resized, or uncovered, the runtime tells that window which region of the screen needs redraw.

Modern graphical systems build upon the 3 above concepts of Inputs, Outputs and Windows, improving and adapting them to suit modern hardware and taste.

II. The X Window System

Fast forward to 1984 for the first appearance of our hero. Not Big Brother nor Steve Job's Macintosh. But indeed the first incarnation of the X Window System.

In the words of the creator of the X Window System, Scheiffler at MIT

I've spent the last couple weeks writing a window system for the VS100. I stole a fair amount of code from W, surrounded it with an asynchronous rather than a synchronous interface, and called it X. Overall performance appears to be about twice that of W... This is not the ultimate window system, but I believe it is a good starting point for experimentation.

The main thrust of X was to provide students with a convenient way to access the university's computing resources. Core to the design was the idea of network transparency: users interact with the X server that runs on a local computer and handles all input/output. Applications, local or remote, connect to the server, receive events from it, and tell the server what to display by sending commands.

The X concept of a "client" and a "server" are somewhat unintuitive as illustrated by the below quote, but this is not so important for the modern consumer use-case where clients and server run on the same machine.

The Unix Haters Handbook Chapter 7. The X-Windows Disaster

We have tried to avoid paragraph-length footnotes in this book, but X has defeated us by switching the meaning of client and server. In all other client/server relationships, the server is the remote machine that runs the application (i.e., the server provides services, such a database service or computation service). For some perverse reason that's better left to the imagination, X insists on calling the program running on the remote machine "the client." This program displays its windows on the "window server." ... when you see "client" think "the remote machine where the application is running," and when you see "server" think "the local machine that displays output and accepts user input."

Worth noting however, is that the network-first architecture of X introduces unnecessary overhead for local clients. These performance issues are mitigated by using Unix domain sockets instead of Internet sockets for commands, and using shared memory for frames. But the complexity of serializing commands and sending them over a network interface still lurks hidden in the implementation of Xlib and XCB.

The X Window System is made of up to 3 parts:

  • The X Display Server: this is the core piece sitting between hardware and software. It handles the 3 building blocks I explained above: inputs followed by events dispatching, screen output, and frame buffer plus screen sharing. On Linux, X11 and Wayland are the only battle-tested display server protocols.
  • A window manager: it gives windows their look through decorations, such as title bars, close buttons, and maximize buttons; and their feel through behaviors such as window placement or tiling. See for example Xfwm (the window manager of the Xfce desktop environment) or i3 (a tiling window manager). From the point of view of the X server, the window manager operates like a regular client application.
  • A compositor: this more recent addition handles effects like transparency and shadows by compositing all windows' frames into a single frame and sending that single frame to the screen. It forces windows to render into an off-screen buffer.

While it is possible to bypass the X Window System on modern Linux with Direct Rendering Manager (DRM), this means seizing direct control over the entire display and inputs, breaking screen sharing and other nice features provided by the window manager, like app switching with Alt-Tab, which is often not what we want.

X reached version 11 in 1987, after a major redesign aiming to make the software compatible with the wild variety of platforms and hardware of the day.

Among others, that release featured an extension system, allowing X to adapt without breaking backward compatibility; and server-side state, relieving client applications from the obligation of sending the whole context in every request. You will see extensions in action later, since they are basically required to write an app that meets modern standards.

X is not the only window system. On Linux there is also Wayland which tries to fix some issues of X11.

A good reason to not pick Wayland as a target even for new apps, is that many folks run X11 (about half of surveyed machines, though the data is sparse). And while X apps work under Wayland (thanks to XWayland), Wayland apps do not work under X.

In other words, X apps run on 100% of Linux machines, whereas Wayland apps run on only about 50% of them.

X has been ported to other OSes, but they usually run their own window system: the Desktop Window Manager (DWM) on Windows, the Quartz Compositor on macOS and SurfaceFlinger on Android.

III. XCB over Xlib

This XCB tutorial explains why XCB is preferred over Xlib

But what is XCB? Xlib has been the standard C binding for the X Window System protocol for many years now. It is an excellent piece of work, but there are applications for which it is not ideal, for example:

  • Small platforms: Xlib is a large piece of code, and it's difficult to make it smaller
  • Latency hiding: Xlib requests requiring a reply are effectively synchronous: they block until the reply appears, whether the result is needed immediately or not.
  • Direct access to the protocol: Xlib does quite a bit of caching, layering, and similar optimizations. While this is normally a feature, it makes it difficult to simply emit specified X protocol requests and process specific responses.
  • Threaded applications: While Xlib does attempt to support multithreading, the API makes this difficult and error-prone.
  • New extensions: The Xlib infrastructure provides limited support for the new creation of X extension client side code.

For these reasons, among others, XCB, an X C binding, has been designed to solve the above problems and thus provide a base for

  • Toolkit implementation.
  • Direct protocol-level programming.
  • Lightweight emulation of commonly used portions of the Xlib API.

The XCB Wikipedia page adds to this picture

The main goals of XCB are to:

  • reduce library size and complexity
  • provide direct access to the X11 protocol

The required size reduction is achieved primarily by restricting XCB's scope to handling the X protocol and omitting Xlib functionality such as its extensive utility library, much of which saw little use by applications. This results in a factor thirty reduction of the compiled library size (as of 2004). Secondary goals include making the C interface asynchronous, facilitating better multithreading and making it easier to implement extensions (via XML protocol descriptions).

The core and extension protocol descriptions are in XML, with a program written in Python creating the C bindings.

A further goal is to be able to use these protocol descriptions to create protocol documentation, more language bindings, and server-side stubs.

This excerpt of another tutorial illustrates the advantages of a non-blocking API

For instance, to lookup a window property, the Xlib code is a single function call:

XGetWindowProperty(dpy, win, atom, 0, 0, False, AnyPropertyType, &type_ret, &format_ret, &num_ret, &bytes_after, &prop_ret);

Xlib generates the request to the X server to retrieve the property and appends it to its buffer of requests. Since this is a request that requires a response, Xlib then flushes the buffer, sending the contents to the X server. Next, Xlib waits until the X server processes all the requests preceding the property retrieve request, and sends the property retrieve reply. Xlib then returns the reply to the client.

...

XCB on the other hand, provides functions generated directly from the protocol descriptions in an "obvious" mechanistic way. XCB functions map directly onto the protocol, with separate functions to put requests into the outgoing buffer and to read results back from the X server asynchronously later. The XCB version of the above code is:

prop_cookie = xcb_get_property (dpy, False, win, atom, XCB_GET_PROPERTY_TYPE_ANY, 0, 0);
prop_reply = xcb_get_property_reply (dpy, prop_cookie, NULL);

The power of XCB is in allowing those two steps to have as much code as you want between them. The programmer decides when to wait for data, instead of being forced to wait for the data returned by a request at the time the request is issued.

To summarize, XCB is a direct wrapper of the X protocol, it is more recent than Xlib, much smaller, and allows for more granular control.

A. Connecting to the X Server

The first thing we need is a connection so we can send requests to the X server.

int main() {
  App app = {0};
  
  // Connect to X11 server
  int preferred_screen = 0;
  xcb_connection_t *connection = xcb_connect(NULL, &preferred_screen);
  int xcb_connect_error_code = xcb_connection_has_error(connection);
  if (xcb_connect_error_code > 0) DIE(connection, xcb_connect_error_to_string(xcb_connect_error_code));

xcb_connect and xcb_disconnect are documented as follows:

/**
 * @brief Connects to the X server.
 * @param displayname The name of the display.
 * @param screenp A pointer to a preferred screen number.
 * @return A newly allocated xcb_connection_t structure.
 *
 * Connects to the X server specified by @p displayname. If @p
 * displayname is @c NULL, uses the value of the DISPLAY environment
 * variable. If a particular screen on that server is preferred, the
 * int pointed to by @p screenp (if not @c NULL) will be set to that
 * screen; otherwise the screen will be set to 0.
 *
 * Always returns a non-NULL pointer to a xcb_connection_t, even on failure.
 * Callers need to use xcb_connection_has_error() to check for failure.
 * When finished, use xcb_disconnect() to close the connection and free
 * the structure.
 */
xcb_connection_t *xcb_connect(const char *displayname, int *screenp);

/**
 * @brief Closes the connection.
 * @param c The connection.
 *
 * Closes the file descriptor and frees all memory associated with the
 * connection @c c. If @p c is @c NULL, nothing is done.
 */
void xcb_disconnect(xcb_connection_t *c);

The first argument of xcb_connect is displayname. It has this format: host:display.screen, for example :0:1. The host part is used to connect to a remote host, display selects the X server instance, and screen selects the screen. Unless you need to connect to a specific display, the right thing to do is to use NULL to get the value from the environment.

As for the second argument screenp, it returns the value of the preferred screen, which can apparently be ignored since 2001 because RandR exposes all monitors as a single screen. I still use it to be compliant with the interface, in case it becomes useful in the future.

The XCB connection hides a queue of pending client requests and a queue of server messages/replies. xcb_connection_t * is passed as an argument to functions that send or receive messages from the X server.

B. Error handling

You can see that the call to xcb_connect is followed by some error handling code. I highly recommend checking for errors to quickly pinpoint issues. xcb_connect_error_to_string is not provided by the XCB library, likewise for any other to string functions. We have to provide these ourselves. DIE is a macro that I defined to exit after printing an error message along with the line and filename at which it occured.

XCB is asynchronous, which means we can send requests, do some work, and then only retrieve the replies. This is exactly what the above snippet does for extensions, atoms, and keyboard mappings. I'll go into more details about each of these at their point of use.

For now, notice that the extension and atoms requests return a cookie (a 32-bit integer), which we can use later to wait for the corresponding reply. xcb_key_symbols_alloc is part of an extension that abstracts some details, and the cookie is stored in xcb_key_symbols_t.

All of these requests take the connection as a parameter because they send some data to the X server. We need to explicitly call xcb_flush to actually send the requests, otherwise they just sit in the buffer.

D. Finding the Preferred Screen & the Visual

  // Find preferred screen
  int screen_i = 0;
  const xcb_setup_t *setup = xcb_get_setup(connection);
  xcb_screen_iterator_t screen_it = xcb_setup_roots_iterator(setup);
  while (screen_it.rem && screen_i < preferred_screen) {
    ++screen_i;
    xcb_screen_next(&screen_it);
  }
  if (screen_i != preferred_screen) DIE(connection, "cannot find preferred screen");
  xcb_screen_t *screen = screen_it.data;

  // Find visual
  xcb_visualid_t visual = 0;
  int depth = 24;
  int visual_class = XCB_VISUAL_CLASS_TRUE_COLOR;
  for (xcb_depth_iterator_t depth_it = xcb_screen_allowed_depths_iterator(screen); depth_it.rem; xcb_depth_next(&depth_it)) {
    if (depth_it.data->depth == depth) {
      for (xcb_visualtype_iterator_t visual_it = xcb_depth_visuals_iterator(depth_it.data); visual_it.rem; xcb_visualtype_next(&visual_it)) {
        if (visual_it.data->_class == visual_class) {
          visual = visual_it.data->visual_id;
          break;
        }
      }
    }
  }
  if (!visual) DIE(connection, "cannot find required visual");

xcb_visualid_t, like most XCB objects, is a handle (a 32-bit integer) to some data stored in the X server. The visual describes the format of pixels. Its depth is the number of bits used for color, 32-bits for transparent windows, otherwise 24. Its class is always set to XCB_VISUAL_CLASS_TRUE_COLOR on modern displays, but historically other values like XCB_VISUAL_CLASS_GRAY_SCALE were used.

The above code retrieves the preferred xcb_screen_t from the metadata stored in the xcb_connection_t, then it iterates over the depths and visuals associated with that screen to find one with depth 24 and class XCB_VISUAL_CLASS_TRUE_COLOR.

XCB iterators are consistently named, but the interface is a bit unusual. Unfortunately, you cannot iterate the data directly because depths, for example, contain a variable number of visuals, which means their size is not fixed so you cannot add the size to get the next depth.

Screens contain useful fields, like their width and height.

E. Creating Cursors

#include <xcb/xcb_cursor.h>
  // Create cursors
  xcb_cursor_t cursors[Cursor_Size] = {0};
  xcb_cursor_context_t *cursor_context = NULL;
  int xcb_cursor_context_new_error = xcb_cursor_context_new(connection, screen, &cursor_context);
  if (xcb_cursor_context_new_error < 0) {
    DIE(connection, strerror(-xcb_cursor_context_new_error));
  }
  for (int cursor_index = 0; cursor_index < Cursor_Size; ++cursor_index) {
    cursors[cursor_index] = xcb_cursor_load_cursor(cursor_context, get_cursor_name(cursor_index));
    if (!cursors[cursor_index]) DIE(connection, "xcb_cursor_load_cursor");
  }
  const xcb_query_extension_reply_t *query_extension_render_reply = xcb_get_extension_data(connection, &xcb_render_id);
  if (!query_extension_render_reply) DIE(connection, "xcb_get_extension_data: render");
  RENDER_MAJOR_OPCODE = query_extension_render_reply->major_opcode;
  xcb_cursor_context_free(cursor_context);

X11 windows inherit their cursor from the parent window, but they inherit only one cursor, usually the default one. You can use vanilla X11 with fonts or bitmaps to get more cursors, but they support only 2 colors: 0 and 1, which means you can't get anti-aliasing. Software rendering is also not an option because it bypasses the platform's cursor theme, and more importantly if the app runs slow the cursor won't move, which is really bad UX.

So, for a modern experience, the xcb_cursor extension is required. The above code creates a context to load cursors with xcb_cursor_context_new. Then it iterates on a Cursor enum I've defined, translates the enum value to a string which it passes to xcb_cursor_load_cursor, and stores the returned handle in an array. Finally it frees the context with xcb_cursor_context_free since it's not needed anymore.

You may also notice the xcb_get_extension_data call, which is used to improve error messages. Under the hood, the xcb_cursor extension loads and sends requests to the RENDER extension. If any of these RENDER requests fail, the X server sends back an error message with a major opcode to identify the extension, and a minor opcode to identify the request. We can then map these codes to strings to get readable error messages. The catch is that unlike major opcodes for core requests, extension major opcodes are allocated dynamically, so we need to retrieve them from the X server. In this case this is accomplished with xcb_get_extension_data, which gets the reply from cache because the xcb_cursor extension already fetched the data. When the reply is not in cache the xcb_get_extension_data sends a request to the server and blocks until it receives a reply.

F. Creating a window

  // Create window
  app.width = screen->width_in_pixels/2;
  app.height = screen->height_in_pixels/2;
  int x = (screen->width_in_pixels - app.width)/2;
  int y = (screen->height_in_pixels - app.height)/2;
  int border_width = 0;
  int class = XCB_WINDOW_CLASS_INPUT_OUTPUT;
  uint32_t value_mask = XCB_CW_BACK_PIXMAP | XCB_CW_BIT_GRAVITY | XCB_CW_EVENT_MASK | XCB_CW_CURSOR;
  xcb_create_window_value_list_t value_list = {0};
  value_list.background_pixmap = XCB_BACK_PIXMAP_NONE;
  value_list.bit_gravity = XCB_GRAVITY_STATIC;
  value_list.event_mask = XCB_EVENT_MASK_KEY_PRESS
                        | XCB_EVENT_MASK_KEY_RELEASE
                        | XCB_EVENT_MASK_BUTTON_PRESS
                        | XCB_EVENT_MASK_BUTTON_RELEASE
                        // | XCB_EVENT_MASK_ENTER_WINDOW
                        // | XCB_EVENT_MASK_LEAVE_WINDOW
                        | XCB_EVENT_MASK_POINTER_MOTION
                        // | XCB_EVENT_MASK_KEYMAP_STATE
                        | XCB_EVENT_MASK_VISIBILITY_CHANGE
                        | XCB_EVENT_MASK_STRUCTURE_NOTIFY
                        // | XCB_EVENT_MASK_FOCUS_CHANGE
                        ;
  value_list.cursor = cursors[app.cursor];
  if (visual != screen->root_visual) {
    value_mask |= XCB_CW_BORDER_PIXEL | XCB_CW_COLORMAP;
    value_list.border_pixel = screen->black_pixel;
    value_list.colormap = xcb_generate_id(connection);
    xcb_create_colormap(connection, XCB_COLORMAP_ALLOC_NONE, value_list.colormap, screen->root, visual);
  }
  xcb_window_t window = xcb_generate_id(connection);
  xcb_create_window_aux(connection, depth, window, screen->root, x, y, app.width, app.height, border_width, class, visual, value_mask, &value_list);

xcb_generate_id locally generates an ID/handle for the window, and xcb_create_window_aux tells the server to create a window and associate it to the provided ID. As usual, the xcb_create_window_aux request takes the connection, then the depth (set in step D), the window ID, the screen's (set in step D) root window ID, the position and dimensions of the window, the border width which is usually ignored, the class which has to be InputOutput in order to receive events and send graphic requests, the visual (retrieved in step D), and finally a value_mask and its associated value_list.

When creating a window, the X server uses default values for every field in the value_list, unless a mask is set in the value_mask, in which case the corresponding value of the value_list is used. For example in the above snippet, the XCB_CW_BACK_PIXMAP mask bit tells the X server to use the XCB_BACK_PIXMAP_NONE value for the window attribute background_pixmap.

XCB_BACK_PIXMAP_NONE tells the X server to not paint areas of the window that are newly exposed, and XCB_GRAVITY_STATIC tells the server to not move pixels when the window is resized. I have found that setting these 2 options helps reduce flicker when resizing.

Next comes XCB_CW_EVENT_MASK, which allows you to specify which categories of events you want to receive in the event_mask value. I have selected all useful events for most apps: keyboard press and release, mouse button press and release, and mouse motion. Visibility change is supposed to be sent when the window is exposed or when another window covers it so that you don't render when the window is hidden, but in my tests the event is not sent when the window is covered, apparently so that window managers can show the app in the app switcher, though it's also possible I missed something. And structure notify is sent when the window is moved or resized.

I also commented out other event masks that may be useful, in order: XCB_EVENT_MASK_ENTER_WINDOW and XCB_EVENT_MASK_LEAVE_WINDOW for when the pointer enters or leaves the window, XCB_EVENT_MASK_KEYMAP_STATE to detect modifiers pressed outside the window upon window entry and XCB_EVENT_MASK_FOCUS_CHANGE to detect when the window gets keyboard focus.

Finally the last value mask (not event mask) is XCB_CW_CURSOR to set the window's cursor.

Note that when the selected visual is not the screen's root visual, a colormap must be created and additional masks must be set, as shown in the above snippet.

The window is created unmapped, which is to say hidden in X terminology.

The above code changes the window title by sending a request to change the XCB_ATOM_WM_NAME property to the string "XCB Software Renderer". format is the size in bits of one element in title. In X11, atoms are 32-bit integer handles to a server-side string. The icon name is used when the window is iconified.

H. Creating the frame buffers

  // Create buffers
  int stride = (screen->width_in_pixels * 4 + 63) & ~63;
  app.stride = stride / 4;

  int page_size = getpagesize();
  int size = (stride * screen->height_in_pixels + (page_size - 1)) & ~(page_size - 1);

  int udmabuf_device = open("/dev/udmabuf", O_RDWR);
  if (udmabuf_device < 0) DIE(connection, strerror(errno));

  uint32_t *buffers[BUFFER_COUNT] = {0};
  int udmabufs[BUFFER_COUNT] = {0};
  xcb_pixmap_t pixmaps[BUFFER_COUNT] = {0};

  for (int i = 0; i < BUFFER_COUNT; ++i) {
    int memfd = memfd_create("buffer", MFD_CLOEXEC | MFD_ALLOW_SEALING);
    if (memfd < 0) DIE(connection, strerror(errno));

    if (ftruncate(memfd, size) < 0)
      DIE(connection, strerror(errno));

    if (fcntl(memfd, F_ADD_SEALS, F_SEAL_SHRINK) == -1)
      DIE(connection, strerror(errno));

    buffers[i] = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, memfd, 0);
    if (buffers[i] == MAP_FAILED) DIE(connection, strerror(errno));
    memset(buffers[i], 0, size);

    struct udmabuf_create create = {0};
    create.memfd = memfd;
    create.size = size;
    create.flags = UDMABUF_FLAGS_CLOEXEC;
    udmabufs[i] = ioctl(udmabuf_device, UDMABUF_CREATE, &create);
    if (udmabufs[i] == -1) DIE(connection, strerror(errno));

    uint8_t bpp = 32;
    pixmaps[i] = xcb_generate_id(connection);
    xcb_dri3_pixmap_from_buffer(connection, pixmaps[i], window, size, screen->width_in_pixels, screen->height_in_pixels, stride, depth, bpp, udmabufs[i]);

    close(memfd);
  }

  close(udmabuf_device);

The pixels an app renders go into frame buffers. Here I create 2 frame buffers in order to avoid visual artifacts (more on that later).

On a 1920x1080 screen, a frame buffer is about 8MB (1920*1080*4 bytes). This is a lot of data to transfer each frame, so in order to spare memory bandwidth, the MIT-SHM extension was invented as soon as 1987. SHM stands for shared memory, and is a System-V mechanism supported by Linux to share memory between 2 processes.

If you look around on the internet for an efficient way to transfer frame buffers to the X server, you will quickly find out about the MIT-SHM extension. But this is misleading: first, the creation of shared frame buffers with the shm* syscalls was supplanted by the sharing of frame buffers using memfd along with a more recent API of the MIT-SHM extension; this fixes some security holes related to shm* syscalls permission issues, and avoids having to cleanup the shared memory segments.

But that's not the end: according to XCB mailing list hackers, the preferred way to share frame buffers is to use the DRI3 extension, even for software rendering. Using DRI3 for a software renderer is made possible by dma buffers, which are a Linux mechanism to share a memory handle everywhere: CPU, GPU and drivers.

I also evaluated using Direct Rendering Manager (DRM) dumb-buffers, which is more-or-less VRAM accessible from the CPU, but the memory is mapped write-combined, so while writes are fast, reads are extremely slow.

Thus, the above code uses the DRI3 path to share frame buffers with X11. First I compute a stride by rounding up to 64 byte, to get enough padding for SIMD rendering. Then I compute a frame buffer's byte size, it has to be divisible by the page size.

I open the udmabuf device so I can create dma buffers, create a memfd which is a file descriptor to memory, give the memfd its size with truncate, and add the required F_SEAL_SHRINK to prevent anyone from shrinking the memfd.

Then I mmap the memfd to get a pointer to the underlying memory, marking the memory shared with MAP_SHARED.

I can now use ioctl with UDMABUF_CREATE to get a dma buffer backed by the memfd's memory.

Finally, I call xcb_generate_id to generate an ID for an xcb_pixmap_t, which is an X11 object we can use with graphic requests. And I give this pixmap to to the DRI3 extension call xcb_dri3_pixmap_from_buffer to create the pixmap on the server and associate it to the dma buffer. bpp stands for bits per pixels.

The code goes through the loop one more time, and the 2 frame buffers are set up.

The idea with Present is to block until we receive the CompleteNotify event, which means the server has presented our frame. But what if there's an error and we don't get a CompleteNotify event? Well, in that case the app hangs. Present doesn't offer an API to time-out on the wait, so I implement my own with the above code, by registering an epoll file descriptor on the X server connection.

M. Handling events

  while (!quit) {
    struct timespec start_time = {0};
    struct timespec end_time = {0};

    if (clock_gettime(CLOCK_MONOTONIC, &start_time) == -1)
      DIE(connection, strerror(errno));

    int xcb_connection_error_code = xcb_connection_has_error(connection);
    if (xcb_connection_error_code > 0) DIE(connection, xcb_connect_error_to_string(xcb_connection_error_code));

    // Handle events
    Events app_events = {0};
    xcb_generic_event_t *event = NULL;
    while ((event = xcb_poll_for_event(connection))) {
      if (event->response_type == 0) {
        xcb_generic_error_t *error = (xcb_generic_error_t *)event;
        xcb_print_error(error);
        DIE(connection, "xcb_print_error");
      } else {
        xcb_print_event(event);
        switch (event->response_type & 0x7F) {
          case XCB_CONFIGURE_NOTIFY: {
            const xcb_configure_notify_event_t *e = (const xcb_configure_notify_event_t *)event;
            app.width = e->width;
            app.height = e->height;
            if (app.width > screen->width_in_pixels) {

              app.width = screen->width_in_pixels;
            }
            if (app.height > screen->height_in_pixels) {
              fprintf(stderr, "[WARNING] cannot resize window to be taller than screen, clamping to screen height\n");
              app.height = screen->height_in_pixels;
            }
            acknowledged_sync_value = pending_sync_value;
            pending_sync_value = zero_sync_value;
          } break;
          case XCB_CLIENT_MESSAGE: {
            const xcb_client_message_event_t *e = (const xcb_client_message_event_t *)event;
            if (e->type == wm_protocols) {
              if(e->data.data32[0] == wm_delete_window) {
                fprintf(stderr, "Received WM_DELETE_WINDOW, quitting...\n");
                quit = true;
              } else if (e->data.data32[0] == wm_sync_request) {
                pending_sync_value.lo = e->data.data32[2];
                pending_sync_value.hi = e->data.data32[3];
              }
            }
          } break;
          case XCB_MAPPING_NOTIFY: {
            xcb_mapping_notify_event_t *e = (xcb_mapping_notify_event_t *)event;
            xcb_refresh_keyboard_mapping(key_symbols, e);
          } break;
          case XCB_KEY_PRESS: {
            xcb_key_press_event_t *e = (xcb_key_press_event_t *)event;
            // Note: blocks if the reply from a mapping refresh has yet to arrive
            xcb_keysym_t keysym = xcb_key_press_lookup_keysym(key_symbols, e, 0);
            if (keysym >= ' ' && keysym <= 255) {
              app_events.key = keysym;
            }
          } break;
          case XCB_BUTTON_PRESS: {
            const xcb_button_press_event_t *e = (const xcb_button_press_event_t *)event;
            app_events.clicked = (e->detail == 1);
          } break;
          case XCB_MOTION_NOTIFY: {
            const xcb_motion_notify_event_t *e = (const xcb_motion_notify_event_t *)event;
            app.mouse_x = e->event_x;
            app.mouse_y = e->event_y;
          } break;
        }
      }
      free(event);
    }

For games and immediate-mode GUI apps, the event-loop structure is as follows:

  • a) drain the event queue to handle all events that occured in the previous frame and store the bits necessary to render
  • b) render
  • c) present frame
  • d) block until VSYNC
  • e) swap buffers
  • repeat

    • The above code does step a).

    The loop exits when we receive an XCB_CLIENT_MESSAGE containing the wm_delete_window atom we registered in step G. This is a client message because the window's close button is handled by the window manager, which is itself a client of the X server.

    First thing in the loop we measure the start time with clock_gettime, CLOCK_MONOTONIC is important to always get a positive delta.

    We check the connection for errors, and then we start polling events with xcb_poll_for_event. Two cases: if the event is an error, we print the error with the custom xcb_print_error function and DIE. Otherwise we handle the event, before freeing it.

    Handling events starts with the custom xcb_print_event, which is very useful for developing and can also be used as a reminder to remember how to get to various fields in the event struct.

    By default, the window flickers on resize. We registered the sync extension with the window manager above to mitigate that. The idea is that instead of brutally resizing our window, the window manager politely sends us a sync counter value through an XCB_CLIENT_MESSAGE containing the wm_sync_request atom, followed by an XCB_CONFIGURE_NOTIFY containing the new target size. This allows us to first render at the new size, and then only notify the window manager that it can resize our window. So in theory we should get butter smooth resize, without any flickering. In practice I got less flickering, but there are still artifacts during resize. For example if you resize the top of the window back and forth, you will see a black band go up and down at the bottom of the window. The sync counter value is used to identify which resize request is being acknowledged.

    Note that the window can theoretically be larger than the screen, though this is mostly useful on multi-monitors setup. For now, I just clamp the window to screen size.

    XCB_BUTTON_PRESS tells you if a mouse button was pressed, and if so which one. 1 is left click. XCB_MOTION_NOTIFY is for mouse motion. XCB_KEY_PRESS contains key codes which should be mapped to keysyms before use to account for modifiers and so on. Keysyms from space to 255 map to ASCII, which is convenient.

    You should actually use the xkb.h extension for modern keyboard handling. I don't show it here because unfortunately I implemented the XCB_KEY_PRESS handling based off stale info, which is something to be wary of with old, open-source projects like X11. Anyway, xkb.h is useful for text-heavy apps, as it has better layout reloading, dead keys handling, and mode handling.

    XCB_MAPPING_NOTIFY is received when the keymap changes, in which case you should reload it. This event is not sent reliably by modern window managers, which is another reason going through xkb.h is preferred.

    void render(App *app, uint32_t *buf, Events* events, int frame, float processing_dt) {
  // Clear
  // draw_rectangle(app, buf, 0, 0, app->width, app->height, app->stride, BLACK);
  app->cursor = Cursor_Default;

  // Tear test
  int stripes_speed = 3;
  int stripes_width = 64;
  for (int y = 0; y < app->height; y++) {
    for (int x = 0; x < app->width; x++) {
      int stripe_pos = (x + app->stripes_frame*stripes_speed) % (stripes_width*2);
      uint32_t color = (stripe_pos < stripes_width) ? COLOR_STRIPE_A : COLOR_STRIPE_B;
      buf[y*app->stride + x] = color;
    }
  }
  int scan_line_y = (app->stripes_frame % 120)*app->height/120;
  draw_rectangle(app, buf, 0, scan_line_y, app->width, 1, app->stride, COLOR_SCAN_LINE);
  if (!app->paused) app->stripes_frame += 1;

  // Frame dt graph
  FRAME_DT_GRAPH[frame % FRAME_DT_GRAPH_SIZE] = processing_dt;

  int frame_base_width = app->width / FRAME_DT_GRAPH_SIZE;
  int rem_width = app->width % FRAME_DT_GRAPH_SIZE;

  int FPS_AT_MAX_HEIGHT = 60;
  int half_height = app->height / 2;

  for (int y = 0; y < half_height; ++y) {
    for (int frame_index=0, frame_start_x=0; frame_index < FRAME_DT_GRAPH_SIZE; ++frame_index) {

      int frame_dt_index = (frame + 1 + frame_index) % FRAME_DT_GRAPH_SIZE;
      int frame_height = FRAME_DT_GRAPH[frame_dt_index]*half_height*FPS_AT_MAX_HEIGHT;
      if (frame_height > half_height) frame_height = half_height;

      int frame_width_with_rem = frame_base_width + ((frame_index < rem_width) ? 1 : 0);

      if (y >= (half_height - frame_height)) {
        for (int x = frame_start_x; x < frame_start_x + frame_width_with_rem - 1; ++x) {
          buf[y*app->stride + x] = COLOR_DT_GRAPH;
        }
      }

      frame_start_x += frame_width_with_rem;
    }
  }

  // Pause Button
  int button_w = 80;
  int button_h = 50;
  int button_x = (app->width - button_w)/2;
  int button_y = (app->height*3 - button_h*2)/4;
  int button_color = COLOR_BUTTON;

  if ((app->mouse_x >= button_x && app->mouse_x < (button_x + button_w)) &&
      (app->mouse_y >= button_y && app->mouse_y < (button_y + button_h))) {
    button_color = COLOR_BUTTON_HOVER;
    app->cursor = Cursor_Pointer;
    if (events->clicked) {
      app->paused = !app->paused;
    }
  }

  draw_rectangle(app, buf, button_x, button_y, button_w, button_h, app->stride, button_color);

  // Pause Key
  if (events->key == ' ') {
    app->paused = !app->paused;
  }
}

    xcb_present_pixmap has lots of arguments, which are mostly set to 0. These tell X to present the frame at the next VSYNC. We use current_buffer to index into the pixmaps array. The X server might still use the buffer even after we received the CompleteNotify event, so to avoid visual artifacts, we use a second buffer we render into while the first one might still be busy.

    In addition to double-buffering, we can listen to another Present event that tells us the buffer is now available, but I decided against implementing this because this is only useful when the X server is still using the first buffer after a full frame passed, which basically never happens, and in the rare cases it does happen, we have two choices: wait for the buffer to be available, which means we'll miss the VSYNC and the user will see stutter, or render into the busy buffer anyway which means the user may see tearing. I don't have much of a preference, so I chose the simpler path which is to not listen for this event.

    After presenting we update the cursor if required, and call xcb_flush, which is a nice reminder that XCB is asynchronous and nothing happens unless we explicitly flush or wait on a reply.

    Then we get the end time which is used for drawing the frame graph.

        break;

        if (errno == EINTR) continue;
        DIE(connection, strerror(errno));
      }
    }
  sync_end:

    if (acknowledged_sync_value.lo != 0 || acknowledged_sync_value.hi != 0) {
      xcb_sync_set_counter(connection, sync_counter, acknowledged_sync_value);
      acknowledged_sync_value = zero_sync_value;

    After presenting the pixmap, we wait for VSYNC so that our app stays in sync with the refresh rate of the monitor.     V. XCB First Impressions

    I found it hard to create a platform layer with XCB. The documentation is incomplete (it contains TODOs) and out-of-date. I didn't find any production-ready examples either.

    You'll need to use extensions to get things done, and you won't know it in advance. I implemented the memory buffer sharing code 3 times, discovering a more modern way each time.

    It feels like everything has to be explicitly set up to get the minimum we expect from a modern app. Having modern defaults might be a better trade-off there. IIRC Win32 has something like that to get default behavior when handling events.

    On the plus side, I like the consistent nature of the naming scheme and API, as well as the direct mapping to the X11 protocol.

    VI. And that's a wrap...

    Finally, we are done setting up XCB. Or are we?

    I noted a few things that I could still improve:

    • use the XKB extension for keyboard handling
    • optimize away rendering when the window is hidden
