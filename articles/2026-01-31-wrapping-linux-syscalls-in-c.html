Wrapping Linux Syscalls in C

Previously: ⬅ Gathering Linux Syscall Numbers in a C Table.

I've been writing a header-only C library so I can issue Linux syscalls without libc on x86, ARM and RISC-V, 32-bits and 64-bits. Last time I detailed the rather tedious process of gathering syscall numbers for several architectures.

This time I'll explain how I wrote C wrappers so I can actually use these syscalls.

To issue a syscall, one has to put the syscall number in a register, place the arguments in other registers, and issue an assembly instruction. Because different CPU architectures use different registers and Instruction Set Architectures (ISA), syscall conventions are architecture-specific.

Actually, using a syscall is not that different from calling a function: function arguments are also passed in registers, and this is architecture-specific too.

System V Application Binary Interface (x86-64)

Classification The size of each argument gets rounded up to eight bytes.14 The basic types are assigned their natural classes:

  • Arguments of types (signed and unsigned) _Bool, char, short, int, long, long long, and pointers are in the INTEGER class.
  • Arguments of types _Float16, float, double, _Decimal32, _Decimal64 and __m64 are in class SSE.

...

Passing Once arguments are classified, the registers get assigned (in left-to-right order) for passing as follows:

  1. If the class is MEMORY, pass the argument on the stack at an address respecting the argument's alignment (which might be more than its natural alignment).
  2. If the class is INTEGER, the next available register of the sequence %rdi, %rsi, %rdx, %rcx, %r8 and %r9 is used.19
  3. If the class is SSE, the next available vector register is used, the registers are taken in the order from %xmm0 to %xmm7.
  4. If the class is SSEUP, the eightbyte is passed in the next available eightbyte chunk of the last used vector register.
  5. If the class is X87, X87UP or COMPLEX_X87, it is passed in memory.

Procedure Call Standard for the Arm 64-bit Architecture (AAPCS64)

6.8 Parameter passing

The base standard provides for passing arguments in general-purpose registers (r0-r7), SIMD/floating-point registers (v0-v7), scalable vector registers (z0-z7, overlaid on v0-v7), scalable predicate registers (p0-p3), and on the stack. For subroutines that take a small number of small parameters, only registers are used.

The main difference between function calls and syscalls is the requirement for a specific assembly instruction (syscall on x86_64, svc #0 on ARM, or ecall on RISC-V), and that is for security reasons: syscalls are a mechanism to ensure that user-space processes hand over control to the kernel's trusted code before accessing resources. The hardware elevates the CPU's privilege level and transfers control to the kernel, which then decides whether to grant resource access.

Back to implementing syscall wrappers. One might reason:

  • Syscalls and function calls are similar: they both pass arguments in registers.
  • The compiler handles the architecture-specific details of function calls.
  • Surely, the compiler must also provide an architecture-independent way of issuing syscalls.

But the connoisseur might have guessed where this is going:

Wrong!

The compiler does not provide a way to use syscalls independent of the architecture. Granted, libc implements a syscall function, but that does not equal a language built-in: libc-free programs are on their own.

As backward as it may sound, there is no way to issue syscalls in standard ISO C. The usual way to get around this limitation is to use assembly, either in separate files linked to a C program or inline via compiler extensions.

Since I am writing a header-only library, separate assembly files are not an option. So inline assembly it is!

While non-standard, the GNU inline assembly extension is supported by both GCC and Clang, which is fine for my use-case1. From the official docs, the syntax seems straightforward.

6.11.2 Extended Asm — Assembler Instructions with C Expression Operands

With extended asm you can read and write C variables from assembler and perform jumps from assembler code to C labels. Extended asm syntax uses colons (‘:’) to delimit the operand parameters after the assembler template:

asm asm-qualifiers ( AssemblerTemplate 
                   : OutputOperands 
                   [ : InputOperands
                   [ : Clobbers ] ])

  asm asm-qualifiers ( AssemblerTemplate 
                        : OutputOperands
                        : InputOperands
                        : Clobbers
                        : GotoLabels)

where in the last form, asm-qualifiers contains goto (and in the first form, not).

The asm keyword is a GNU extension. When writing code that can be compiled with -ansi and the various -std options, use __asm__ instead of asm...

But reading further along, I was dismayed to see that there are a number of gotchas. For example:

Extended Asm Documentation — 6.11.2.1 Volatile

GCC’s optimizers sometimes discard asm statements if they determine there is no need for the output variables. Also, the optimizers may move code out of loops if they believe that the code will always return the same result... Using the volatile qualifier disables these optimizations. asm statements that have no output operands and asm goto statements, are implicitly volatile.

And in case you thought using volatile would get you out of trouble, a second layer is laid out to catch even the most seasoned of programmers:

Extended Asm Documentation — 6.11.2.1 Volatile (Continued)

Note that the compiler can move even volatile asm instructions relative to other code, including across jump instructions. For example, on many targets there is a system register that controls the rounding mode of floating-point operations. Setting it with a volatile asm statement, as in the following PowerPC example, does not work reliably.

asm volatile("mtfsf 255, %0" : : "f" (fpenv));
sum = x + y;

The compiler may move the addition back before the volatile asm statement. To make it work as expected, add an artificial dependency to the asm by referencing a variable in the subsequent code, for example:

asm volatile ("mtfsf 255,%1" : "=X" (sum) : "f" (fpenv));
sum = x + y;

As far as I can tell, the agreed-upon way to avoid instruction reordering is to use memory clobber in addition to volatile2.

Extended Asm Documentation — 6.11.2.6 Clobbers and Scratch Registers

"memory"
The "memory" clobber tells the compiler that the assembly code performs memory reads or writes to items other than those listed in the input and output operands (for example, accessing the memory pointed to by one of the input parameters). To ensure memory contains correct values, GCC may need to flush specific register values to memory before executing the asm. Further, the compiler does not assume that any values read from memory before an asm remain unchanged after that asm; it reloads them as needed. Using the "memory" clobber effectively forms a read/write memory barrier for the compiler.

Note that this clobber does not prevent the processor from doing speculative reads past the asm statement. To prevent that, you need processor-specific fence instructions.

Let's illustrate the inline assembly syntax with a syscall wrapper.

_Syscall6_linux implementation for x86 64-bits

long _Syscall6_linux(long number, long a, long b, long c, long d, long e, long f, long* ret2) {
  register long rax __asm__("rax") = number;
  register long rdi __asm__("rdi") = a;
  register long rsi __asm__("rsi") = b;
  register long rdx __asm__("rdx") = c;
  register long r10 __asm__("r10") = d;
  register long r8  __asm__("r8")  = e;
  register long r9  __asm__("r9")  = f;

  __asm__ volatile (
    "syscall" // AssemblerTemplate 
    : "=r" (rax), "=r" (rdx) //  OutputOperands
    : "r" (rax), "r" (rdi), "r" (rsi), "r" (rdx), "r" (r10), "r" (r8), "r" (r9) // InputOperands
    : "rcx", "r11", "memory" // Clobbers
  );

  if (ret2) {
    *ret2 = rdx;
  }
  return rax;
}

Extended Asm Documentation — 6.11.6.2 Local Register Variables

You can define a local register variable and associate it with a specified register like this:

register int *foo asm ("r12");

Do not use type qualifiers such as const and volatile, as the outcome may be contrary to expectations... be aware that a register (for example r0) can be call-clobbered by subsequent code

I use the register keyword to bind variables to registers. While GCC allows specifying register constraints directly in the asm block on x86, this does not work on ARM and RISC-V where constraints are less granular.

Constraints for asm operands — 6.11.3.1 Simple Constraints

'r'
A register operand is allowed provided that it is in a general register.

Constraints for asm operands — 6.11.3.3 Constraint Modifier Characters

'='
Means that this operand is written to by this instruction: the previous value is discarded and replaced by new data.

Note that in the above syscall wrapper rax serves both as an input (the syscall number, with a generic "r" constraint) and as an output (the result, with a generic "=r" constraint).

Another puzzle was identifying the clobbered registers. There are some hints in the comments of entry.S files in the kernel, for example this comment for x86_64.

Comment in linux/arch/x86_64/entry.S

* 64-bit SYSCALL saves rip to rcx, clears rflags.RF, then saves rflags to r11,

But cross-referencing clobbered registers with battle-tested implementations like musl or glibc is easier.

Some syscalls3 return two values in two registers. I chose an API that returns the second value via a pointer (ret2) to stay compatible across all architectures.

To make the API a bit easier to use, I also defined a macro that handles the casting automatically.

Syscall6_linux macro

#define Syscall6_linux(number, a, b, c, d, e, f, ret2) _Syscall6_linux(number, (long)(a), (long)(b), (long)(c), (long)(d), (long)(e), (long)(f), (long*)(ret2))

_Syscall0_linux to _Syscall5_linux have similar implementations, with fewer arguments. Although some syscalls on some architectures require a Syscall7, I did not need it to support all modern syscalls on x86, ARM and RISC-V in both 32-bits & 64-bits. For completeness, here are the implementations of _Syscall6_linux on ARM and RISC-V 64 bits, you can notice the syscall instruction, register names, and clobbered registers change.

_Syscall6_linux implementation on ARM 64-bits

long _Syscall6_linux(long number, long a, long b, long c, long d, long e, long f, long* ret2) {
    register long x8 __asm__("x8") = number;
    register long x0 __asm__("x0") = a;
    register long x1 __asm__("x1") = b;
    register long x2 __asm__("x2") = c;
    register long x3 __asm__("x3") = d;
    register long x4 __asm__("x4") = e;
    register long x5 __asm__("x5") = f;
    __asm__ volatile (
      "svc #0"
      : "=r" (x0), "=r" (x1)
      : "r" (x8), "r" (x0), "r" (x1), "r" (x2), "r" (x3), "r" (x4), "r" (x5)
      : "memory", "cc"
    );
    if (ret2) {
      *ret2 = x1;
    }
    return x0;
  }

_Syscall6_linux implementation on RISC-V

long _Syscall6_linux(long number, long a, long b, long c, long d, long e, long f, long* ret2) {
    register long a7 __asm__("a7") = number;
    register long a0 __asm__("a0") = a;
    register long a1 __asm__("a1") = b;
    register long a2 __asm__("a2") = c;
    register long a3 __asm__("a3") = d;
    register long a4 __asm__("a4") = e;
    register long a5 __asm__("a5") = f;
    __asm__ volatile (
      "ecall"
      : "=r" (a0), "=r" (a1)
      : "r" (a7), "r" (a0), "r" (a1), "r" (a2), "r" (a3), "r" (a4), "r" (a5)
      : "memory"
    );
    if (ret2) {
      *ret2 = a1;
    }
    return a0;
  }

And that's it for the implementation of the untyped cross-architecture Syscall API.

A quick usage demo

#define C_LINUX_IMPLEMENTATION
#include "linux.h"

__attribute__((force_align_arg_pointer))
void main(void) {
  int STDOUT = 1;
  char* message = "Hello, libc-free world!\n";
  int messageSize = 24;
  long* ignoreRet2 = 0;
  Syscall3_linux(NR_write_linux, STDOUT, message, messageSize, ignoreRet2);

  int exitCode = 42;
  Syscall1_linux(NR_exit_linux, exitCode, ignoreRet2);
}

$ clang -O0 -nostdlib -static -fuse-ld=lld -ffreestanding -o linux_demo linux_demo.c -e main && ./linux_demo
Hello, libc-free world!

Edit: Thank you to the Redditor pointing out that the stack pointer must be aligned to 16 bytes, at least on x86_64. I edited the code sample to do so with force_align_arg_pointer, though creating an assembly entry point that calls main is the more common way to ensure proper stack pointer alignment.

The gist of the issue is that on x86_64 the compiler expects functions such as main to be called from a 16-byte aligned stack, i.e. with the stack pointer's hex address ending in 0. On x86_64 the call instruction then pushes an 8-byte return address on the stack, so the compiler assumes the stack address ends with 8 on function entry, and it emits an instruction to push rbp on the stack, which aligns the stack back to 16.

However Linux transfers control to the entry point without executing a call instruction, so there is no return address pushed on the stack and thus the stack is 16-byte aligned on entry, which breaks the compiler's assumptions if the entry point is a regular function (naked functions and assembly symbols are fine). The actual value of the stack pointer is only known at runtime, so of course the compiler has no way to know in advance that the stack will be unaligned. The compiled assembly proceeds as usual, except it runs with a 16-byte aligned stack instead of the expected 8, and so it pushes rbp on the stack, aligning the stack pointer to 8 bytes before the execution goes on in the function code proper.

This results in a crash on instructions that expect 16-byte aligned memory, like _mm_load_si128 loading a (now 8-byte aligned) stack-allocated array. Note that x86_32 is also affected by this issue, with the difference that the pushed return address is 4 bytes. However ARM and RISC-V are not affected, because they use registers to store the return address, so the stack alignment doesn't change whether call is executed or not.
